# ユースケース記述: 部屋を予約する

## 概要

| 項目 | 内容 |
| --- | --- |
| ユースケース名 | 部屋を予約する |
| 主アクター | 利用者 |
| 関係者 | なし（セルフサービス前提。ホテルスタッフはシステムに直接関与しない） |
| 目的 | 利用者が希望する宿泊条件で空室を探し、予約番号を受け取る |
| 事前条件 | HRSが利用可能であり、予約対象の部屋情報が登録されている |
| 事後条件 | 予約が成立し、HRSに予約情報と予約番号が記録され、利用者に予約確認メールが送信されている |
| 失敗時の事後条件 | 予約情報は作成されず、利用者に予約できない理由が示されている |

## 基本系列

1. 利用者が宿泊日、泊数、人数、部屋タイプを入力する。
2. HRSが入力条件の妥当性を確認する。
3. HRSが条件に合う空室を検索する。
4. HRSが予約可能な部屋候補と宿泊料金を表示する。
5. 利用者が予約する部屋を選択する。
6. HRSが選択された部屋の最新の空室状況を確認する。
7. HRSが利用者情報の入力画面を表示する。
8. 利用者が氏名、メールアドレス（必須）・電話番号（任意）、宿泊者情報を入力する。
9. HRSが入力された利用者情報の妥当性を確認する。
10. HRSが予約内容の確認画面を表示する。
11. 利用者が予約内容を確認し、予約確定を指示する。
12. HRSが予約情報を登録する。
13. HRSが予約番号を発行する。
14. HRSが予約番号と予約内容を表示する。
15. HRSが利用者のメールアドレスに予約確認メールを送信する。

## 代替系列

### A1: 利用者が検索条件を変更する

4a. 利用者が表示された部屋候補を選ばず、検索条件の変更を指示する。
4b. HRSが検索条件を変更できる画面を表示する。
4c. 利用者が検索条件を変更する。
4d. ユースケースは基本系列3に戻る。

### A2: 利用者が部屋選択を修正する

10a. 利用者が予約内容の確認時に部屋選択の修正を指示する。
10b. HRSが部屋候補を表示する。
10c. ユースケースは基本系列5に戻る。

### A3: 利用者が利用者情報を修正する

10a. 利用者が予約内容の確認時に利用者情報の修正を指示する。
10b. HRSが利用者情報の入力画面を表示する。
10c. ユースケースは基本系列8に戻る。

### A4: 利用者が予約を取り消す

任意の入力または確認時点で、利用者が予約操作の中止を指示する。  
HRSは予約情報を作成せず、予約操作を終了する。

## 例外系列

### E1: 入力条件が不正である

2a. HRSが宿泊日、泊数、人数、部屋タイプのいずれかに不正な値を検出する。  
2b. HRSが不正な項目と修正方法を表示する。  
2c. ユースケースは基本系列1に戻る。

### E2: 条件に合う空室がない

3a. HRSが条件に合う空室を見つけられない。
3b. HRSが空室がないことを表示し、条件変更を促す。
3c. 利用者が条件変更を指示した場合、利用者が検索条件を変更し、ユースケースは基本系列3に戻る。
3d. 利用者が中止を指示した場合、HRSは予約情報を作成せず、予約操作を終了する。

### E3: 選択した部屋が予約できなくなっている

6a. HRSが選択された部屋を予約できない状態であると検出する。  
6b. HRSが予約できない理由を表示する。  
6c. HRSが最新の空室候補を表示する。  
6d. ユースケースは基本系列5に戻る。

### E4: 利用者情報が不正である

9a. HRSが氏名、メールアドレス、電話番号、宿泊者情報のいずれかに不正な値を検出する。  
9b. HRSが不正な項目と修正方法を表示する。  
9c. ユースケースは基本系列8に戻る。

### E5: 予約登録に失敗する

12a. HRSが予約情報を登録できない。  
12b. HRSが予約を完了できないことを表示する。  
12c. HRSは予約番号を発行せず、予約操作を終了する。

## 補足

- 連絡先は**メールアドレス（必須）**と**電話番号（任意）**の2フィールドで入力する。メールアドレスは予約確認メール・リマインダー・領収メールの送信先として使用する。
- 予約確認メール（基本系列15）はメール送信サービスの障害時もメインの予約処理を妨げない非同期送信で行う。送信が失敗した場合でも予約は成立している。
- チェックイン・予約確認・キャンセルでは予約番号と宿泊代表者の氏名（姓・名）による本人照合を行う（詳細は各 UC 参照）。
