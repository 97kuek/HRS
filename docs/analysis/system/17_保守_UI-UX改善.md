# 保守: フォームのUI/UX改善

- 対象 Issue: #17（[設計・実装] 保守 (バグ修正・改善)）
- 関連実装: #16（Webアプリケーションの実装）
- 担当: 97kuek

## 1. 概要

- 対象: 利用者向け画面（予約・チェックイン・チェックアウト・キャンセル）の入力体験（UI/UX）
- 変更範囲: プレゼンテーション層のみ。機能（API・ドメインロジック）は変更しない
- 技術スタック: Next.js / TypeScript / Prisma / PostgreSQL（既存に従う）

## 2. 改善の内容

利用者からの要望を「入力フォームのストレス軽減」と「おもてなし（待ち時間・比較・共有）」に整理して反映した。

### 2.1 入力フォームの改善

| 観点 | 対応 |
| --- | --- |
| 半角/全角・ハイフンの明示 | 連絡先・予約番号・部屋番号に入力例つきのヒント文を表示 |
| 入力例・注意事項の表示 | `field-hint` を各フィールド直下に表示 |
| 必須／任意の区別 | ラベルにバッジ表示（`必須` / `任意`） |
| 入力完了率の表示 | `CompletionMeter` で必須項目の充足状況を可視化 |
| 即時フィードバック | `onBlur` で検証し、枠色（`is-valid` / `is-invalid`）で状態を表示 |
| 具体的なエラー理由 | `src/lib/validation.ts` に集約し、例つきメッセージを返す |
| 送信後の次アクション | 予約完了画面に「次にできること」と予約番号コピーを追加 |

### 2.2 おもてなしの改善

| 観点 | 対応 |
| --- | --- |
| 待ち時間に応じた表示 | `LoadingIndicator`：〜4秒はスピナー、4秒以上はプログレスバー |
| 重要画面からの離脱確認 | 予約入力途中のタブ閉じ警告＋「予約を中止」確認モーダル |
| 比較のしやすさ | 空室一覧に並び替え・「気になる★」・残室バッジを追加 |
| 共有のしやすさ | 検索条件をクリップボードにコピーする導線を追加 |

## 3. 要求分析・システム分析の差分

- ユースケース図・ユースケース記述・コラボレーション図: 変更なし
  - UC の系列・アクター・振る舞いを変更していない
  - 改善は画面の表示・入力補助に限定している
- クラス図・ドメインモデル: 変更なし
  - 新規のエンティティ・属性・関連は追加していない
  - 入力補助・ローディング表示はバウンダリの責務であり、モデル化対象外である

## 4. アーキテクチャ設計・実装への影響

変更はプレゼンテーション層に閉じており、アプリケーション層・ドメイン・共通基盤には及ばない。

| 層 | 変更 |
| --- | --- |
| プレゼンテーション層 | 4画面の入力UX刷新、共通UIコンポーネント・CSSユーティリティ追加 |
| アプリケーション層 | 変更なし |
| ドメイン（Prisma schema） | 変更なし |
| 共通基盤（APIエラー定義） | 変更なし |

追加ファイル:

- `src/lib/validation.ts` … 入力検証の純関数（理由・入力例つきメッセージ）
- `src/components/LoadingIndicator.tsx` … 待ち時間に応じたローディング表示
- `src/components/CompletionMeter.tsx` … 入力完了率メーター
- `src/components/LeaveGuard.tsx` … 離脱確認（`beforeunload` フック＋確認モーダル）

変更ファイル:

- `src/app/globals.css` … ヒント・必須/任意バッジ・検証状態・スピナー・プログレスバー・完了メーター・モーダルのスタイル
- `src/app/reservations/new/page.tsx` … 予約フローの入力 UX 刷新・空室一覧の比較機能
- `src/app/check-in/page.tsx` / `src/app/check-out/page.tsx` / `src/app/reservations/cancel/page.tsx` … 入力補助・即時検証・待ち時間対応

## 5. 既存設計との整合

- [バリデーション・エラー設計](../../design/validation-error-design.md): 即時検証・フィールド直下エラーは「UI 検証は利便性のため」「フィールド近くに表示」の方針に沿う。API 側の検証責務は変更していない
- [画面構成・ルーティング設計](../../design/ui-routing-design.md): 送信中のローディング表示への切り替えは「送信中はボタン無効化で二重送信防止」の方針に沿う

## 6. 部屋数・客室画像の扱い（設計整合の確認）

- **部屋数の追加**（`prisma/seed.mjs`）: スキーマ変更なし。部屋タイプ 4 種・客室 10 室に増やした。DB 設計・クラス図と整合している
- **客室画像**: `room_types` に画像属性は定義されていない。DB への追加はドメインモデル・DB 設計の変更となるため、画面側の装飾プレースホルダに留めた。実データとして扱う場合は別 Issue で起票する

## 7. 動作確認

| 確認項目 | 結果 |
| -------- | ---- |
| `npm run lint` / `npm run build`（Docker Postgres 開発 DB） | 通過 |
| 主要画面の表示・即時検証・ローディング表示・空室一覧の並び替え | 確認済み |

## 8. 考察

- 変更がプレゼンテーション層に閉じ、ドメインモデルと共通基盤に波及しなかった
- 入力補助・待ち時間表示・比較機能はすべてバウンダリの責務として実装できた
- 安定した下位層（ドメイン・API）を変更せずに利用者体験を高められた
- 層間の依存方向が保たれていることの一例である
