# 認証・認可設計

- 対象 Issue: [#15](https://github.com/97kuek/HRS/issues/15)
- 前提: [技術選定の記録](../tech-stack/README.md), [API設計](api-design.md), [DB設計](db-design.md)
- 状態: **ドラフト**（[#14 システム分析レビュー](https://github.com/97kuek/HRS/issues/14) 後に見直す）

HRS の現行スコープでは、利用者が予約、予約確認、チェックイン、チェックアウトをセルフサービスで行う。要求分析上のアクターは利用者のみで、管理者、受付係、会員ユーザーはまだ登場していない。

そのため、初期実装では利用者アカウントとログイン機能は導入せず、予約番号と予約者の連絡先による軽量な本人確認を採用する。認可は「本人確認できた予約だけを参照・操作できる」ことを基本ルールとする。

## 結論

| 項目 | 方針 |
| --- | --- |
| 認証方式 | 利用者アカウントなし。予約番号と予約時の連絡先で予約本人を確認する |
| 認可方式 | 予約単位のアクセス制御。確認済みの予約以外は参照・更新できない |
| セッション | 初期実装では使わない。保護対象操作ごとに予約番号と連絡先を `POST` 本文で送る |
| 管理者・受付係 | 現行スコープ外。追加する場合はユーザーアカウントとロールベース認可を別途設計する |
| パスワード保存 | 初期実装では扱わない |

## 用語

| 用語 | この設計での意味 |
| --- | --- |
| 認証 | 操作している人が、その予約の本人とみなせるか確認すること |
| 認可 | 確認済みの予約に対して、その操作を許可するか判定すること |
| 利用者 | 予約する人、宿泊する人。ログインアカウントではなく、`guests` に保存される予約者情報を指す |
| 予約アクセス | 予約番号と連絡先が一致した状態で、その予約に限って参照・操作できる権限 |

## 検討案

### 案A: 予約番号のみ

予約番号を知っている人は予約を確認・操作できる方式。

| 観点 | 内容 |
| --- | --- |
| メリット | 実装が最も簡単。既存の API と DB 設計をほぼ変更しない。ログイン画面や認証テーブルが不要 |
| デメリット | 予約番号が漏れると第三者が予約内容を見られる。チェックインやチェックアウトも実行できてしまう。本人確認として弱い |
| 評価 | 採用しない。学習用の簡易実装としては可能だが、予約内容には氏名・連絡先が含まれるため不十分 |

### 案B: 予約番号 + 予約者連絡先

予約番号と、予約時に入力した連絡先が一致した場合だけ予約を参照・操作できる方式。

| 観点 | 内容 |
| --- | --- |
| メリット | 既存の `guests.contact` を使える。利用者アカウント、パスワード、メール認証を持たずに本人確認を強化できる。現行ユースケースと矛盾しない |
| デメリット | 厳密なログインではない。予約ごとに入力が必要になる。予約番号と連絡先の両方が漏れた場合は防げない。総当たり対策が必要 |
| 評価 | **初期実装で採用する**。現行スコープに対して実装量と安全性のバランスがよい |

### 案C: 利用者アカウント + セッション

メールアドレスとパスワード、または外部ログインで利用者アカウントを作り、ログイン後に予約を操作する方式。実装する場合は Better Auth または Auth.js などの認証ライブラリを使い、Postgres/Prisma にユーザーとセッションを保存する。

| 観点 | 内容 |
| --- | --- |
| メリット | 一般的な Web アプリの認証方式。複数予約の一覧表示、予約履歴、プロフィール管理を実装しやすい。将来の権限管理と接続しやすい |
| デメリット | 会員登録、ログイン、ログアウト、パスワードリセット、メール確認などの要件が増える。`guests` とログインユーザーの関係を再設計する必要がある |
| 評価 | 初期実装では採用しない。会員機能を正式ユースケースに追加する場合に再検討する |

### 案D: 管理者・受付係アカウント + RBAC

ホテル側の担当者や管理者をユーザーとして登録し、ロールで操作範囲を分ける方式。

| 観点 | 内容 |
| --- | --- |
| メリット | 客室管理、予約一覧、料金変更、手動チェックインなどの管理機能を安全に分けられる |
| デメリット | 現行の要求分析に管理者・受付係アクターがない。画面、API、DB、監査ログの設計が増える |
| 評価 | 現行スコープ外。ホテル側管理機能を追加する場合に採用候補とする |

### 案E: 外部認証サービス

Clerk、Auth0、Supabase Auth などの外部サービスに認証を任せる方式。

| 観点 | 内容 |
| --- | --- |
| メリット | 認証まわりの実装を減らせる。多要素認証や外部ログインに対応しやすい |
| デメリット | 外部サービスの設定、環境変数、料金、ベンダーロックインが増える。授業課題の小規模実装には重い |
| 評価 | 初期実装では採用しない。公開運用や本格的なユーザー管理が必要になった場合に再検討する |

## 初期実装の認証フロー

1. 利用者は予約番号と連絡先を入力する。
2. Route Handler は入力形式を検証する。
3. `ReservationAccessService` は予約番号で予約を取得し、予約に紐づく `guests.contact` と入力された連絡先を照合する。
4. 一致した場合だけ、予約詳細の参照、チェックイン、チェックアウト処理へ進む。
5. 一致しない場合は `401 Unauthorized` を返す。

連絡先はメールアドレスまたは電話番号を想定する。照合時は前後空白を除去し、メールアドレスは小文字化する。電話番号はハイフンや空白の扱いを実装時に統一する。

## 初期実装の認可ルール

| 操作 | 認可ルール |
| --- | --- |
| 部屋タイプ一覧取得 | 認証不要 |
| 空室確認 | 認証不要 |
| 予約作成 | 認証不要。ただし入力された連絡先を予約本人確認に使う |
| 予約確認 | 予約番号と連絡先が一致した予約だけ許可 |
| チェックイン | 予約番号と連絡先が一致し、予約状態が `RESERVED` の場合だけ許可 |
| チェックアウト | 予約番号と連絡先が一致し、予約状態が `CHECKED_IN` の場合だけ許可 |

認可は UI 側の表示制御だけに置かず、Route Handler またはアプリケーション層で必ず判定する。該当する許可ルールがない場合は拒否する。

## API への反映方針

- 予約確認、チェックイン、チェックアウトでは、予約番号に加えて連絡先を入力させる。
- 連絡先は URL クエリに載せず、初期実装では `POST` リクエスト本文で扱う。
- 予約確認は、本人確認情報を `POST` 本文で扱えるように `POST /api/reservations/lookup` とする。
- チェックアウトは、利用者に内部IDを直接操作させないため `POST /api/reservations/{reservationNumber}/check-out` とする。
- 認証失敗は `401 Unauthorized`、本人確認済みだが対象予約への操作権限がない場合は `403 Forbidden`、予約状態による拒否は `409 Conflict` とする。

### 推奨 API 形状

| メソッド | パス | 本人確認 | 用途 |
| --- | --- | --- | --- |
| `POST` | `/api/reservations/lookup` | リクエスト本文の `reservationNumber`, `contact` | 予約内容を確認する |
| `POST` | `/api/reservations/{reservationNumber}/check-in` | リクエスト本文の `contact` | 予約をチェックイン済みにする |
| `POST` | `/api/reservations/{reservationNumber}/check-out` | リクエスト本文の `contact` | 宿泊を終了し、支払いを記録する |

`/api/reservations/lookup` のリクエスト例:

```json
{
  "reservationNumber": "HRS-20260703-0001",
  "contact": "taro@example.com"
}
```

`/api/reservations/{reservationNumber}/check-in` のリクエスト例:

```json
{
  "contact": "taro@example.com",
  "checkedInAt": "2026-07-03T15:00:00+09:00"
}
```

`/api/reservations/{reservationNumber}/check-out` のリクエスト例:

```json
{
  "contact": "taro@example.com",
  "checkedOutAt": "2026-07-05T10:00:00+09:00",
  "payment": {
    "amount": 24000,
    "method": "card"
  }
}
```

## DB への反映方針

初期実装では、認証専用の `users`、`sessions`、`roles` テーブルは追加しない。`guests.contact` を予約本人確認に使う。

将来、利用者アカウントや管理者アカウントを導入する場合は、少なくとも次の追加設計が必要になる。

| 追加要素 | 目的 |
| --- | --- |
| `users` | ログイン主体を保存する |
| `accounts` | 外部ログインや複数認証方式と紐づける |
| `sessions` | ログイン状態を保存し、失効できるようにする |
| `roles` または `user_role` | 管理者、受付係、利用者などの権限を表す |
| `guests.user_id` または `reservations.user_id` | 予約者情報とログインユーザーを関連づける |

セッション方式を採用する場合は、ブラウザに保存する値へ氏名、連絡先、支払情報などの個人情報を入れない。Cookie を使う場合は `HttpOnly`、`Secure`、`SameSite` を明示する。

## モジュール案

| モジュール | 配置候補 | 責務 |
| --- | --- | --- |
| `ReservationAccessService` | `features/reservation/application/` | 予約番号と連絡先を照合し、予約アクセス可否を判定する |
| `AuthorizationService` | `lib/application/` または各 feature | 操作対象の予約と確認済み予約が一致しているか、状態が操作可能かを判定する |
| `ReservationAccessForm` | `components/` または `features/reservation/` | 予約番号と連絡先を入力するフォーム |

## 実装時の注意

- 予約番号は推測しにくい形式にする。連番だけにしない。
- 予約番号と連絡先の照合失敗時は、予約番号が存在するかどうかを画面メッセージで区別しない。
- 連続失敗に対しては、同一 IP または同一予約番号への試行回数制限を検討する。
- 予約詳細レスポンスには、画面表示に不要な内部IDや個人情報を含めない。
- チェックイン、チェックアウトなど状態を変える API では、本人確認後も予約状態の業務ルールを必ず検証する。
