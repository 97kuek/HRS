# 技術選定の記録

- [Issue #23](https://github.com/97kuek/HRS/issues/23) の成果物
- HRSをWebアプリケーションとして実装するための技術スタックと主要な設計方針を記録する。

## 技術選定案

| 項目 | 選定 (案) | 補足 |
| --- | --- | --- |
| 言語 | **TypeScript** | フロント/バックを1言語に統一 |
| フレームワーク | **Next.js (App Router)** | UIとAPIを1リポジトリで完結。フルスタック構成 |
| API設計 | **REST風 (Route Handlers)** | `app/api/**/route.ts` でエンドポイントを実装 |
| ORM | **Prisma** | スキーマ駆動・型安全・マイグレーション管理 |
| DB | **Postgres** | デプロイ想定のため採用 (後述) |

## 選定理由

### 言語: TypeScript

- フロントエンドとバックエンドを **1言語に統一**でき、学習・分担コストを抑えられる。
- 型でドメインモデル (予約・部屋・宿泊客など) を表現でき、UML分析設計との対応を取りやすい。

### フレームワーク: Next.js (App Router)

- `App Router` の`Route Handlers` で `REST API` を、同一プロジェクトで UI を実装できる。
- バックエンドを別サーバーとして立てる必要がなく、構成管理がシンプル。
- したがって、`Next.js`1本のフルスタックで実装できる。
- 後続のIssue #15（アーキテクチャ設計）もこの前提で進めることが出来る。

### API設計: REST風

- `GraphQL` は今回のような小規模アプリには重く、学習コストが高い
- ホテル予約規模のドメインには`REST`で十分だと考えた
- システム分析の成果物 (分析クラス・操作) からエンドポイントへ落とし込みやすい。

### ORM: Prisma

- `schema.prisma` が **DB設計の単一の真実** になり、Issue #23 の「DB設計の方針」成果物と直結する。
- 型安全なクエリとマイグレーション管理により、チームでスキーマ変更を共有しやすい。

### DB: Postgres

- ローカル開発・デモ用とでは、`SQLite`が最適である（セットアップが軽い）が、`Vercel`等のサーバレスにデプロイする場合、ファイルベースの`SQLite`は永続されず実質使えない。
- レポート等でデプロイすることを想定するため、`Postgres`を採用する。
